Stop Writing SQL Scripts to Clone Your Databricks Catalogs

How we built Clone-Xs: an open-source toolkit that clones, syncs, and governs Unity Catalog — with 56 CLI commands, a 31-page Web UI, and zero manual SQL

VP
Viral Patel
Data & Platform Engineering · 18 min read

Every data platform team has that one notebook. The one with 200 lines of handwritten SQL that clones a Unity Catalog catalog by running CREATE TABLE ... CLONE for each table, one at a time. It takes 3 hours to run. It silently drops permissions, tags, and ownership. Nobody knows when it last succeeded. And when it fails at table #147 out of 300, you start over from scratch.

We have all been there. The consequences compound: dev environments drift from production, DR copies are weeks stale, CI pipelines cannot create isolated test catalogs, and compliance audits reveal that nobody can answer "who cloned what, when?" The manual approach does not just waste time. It creates risk. Every hand-rolled clone script is a ticking time bomb of broken permissions, missing tables, and invisible drift. We built Clone-Xs because we got tired of defusing them.

Four Scenarios Where Manual Cloning Falls Apart

1. Dev Environment Refresh

Data engineers need fresh copies of production catalogs in their development workspaces, ideally every week. The typical workflow looks like this: someone runs a notebook that lists every schema, then iterates through every table, executing CREATE TABLE ... DEEP CLONE on each one. On a catalog with 24 schemas and 166 tables, this takes the better part of a morning. But the real problem is not speed — it is fidelity. The script copies the data, but it does not copy the permissions. It does not preserve ownership. It does not carry over tags, comments, or column-level security policies. By the time a developer opens the cloned catalog, it is a structurally incomplete facsimile of production. Schema drift goes undetected because nobody is comparing the source and destination after the clone. New columns added in production last Tuesday? They might or might not be there, depending on whether the clone ran before or after the migration. And when an engineer asks "is my dev catalog current?" the honest answer is usually "we don't know."

2. Disaster Recovery

DR copies should be nightly. In practice, they are monthly at best, because the clone script is fragile and nobody wants to maintain it. The script was written by an engineer who left the team six months ago. It hardcodes warehouse IDs. It has no retry logic, so a single transient API error at 2 AM causes the entire job to fail silently. The on-call engineer discovers the failure three days later when someone asks about the DR catalog. There is no alerting, no validation step to confirm that the destination catalog actually matches the source, and no rollback mechanism if the clone produces a corrupt result. The consequence is stark: when an actual disaster strikes, the DR catalog is weeks stale, and the team spends the crisis manually re-cloning tables under pressure — exactly the scenario DR was supposed to prevent.

3. CI/CD Pipelines

Modern data engineering teams want the same workflow that software engineers take for granted: every pull request gets an isolated environment, tests run against real data, and the environment is torn down after merge. In Unity Catalog terms, that means creating a fresh catalog clone for each PR, running integration tests against it, and then dropping it. Manual cloning cannot scale to this. You cannot have a human run a notebook for every PR. You need a CLI command that can be embedded in a GitHub Action or Azure DevOps pipeline, that runs in under a minute, that creates a named clone with a TTL (time to live), and that automatically cleans itself up. Without this, teams compromise: they share a single "test" catalog across all PRs, tests interfere with each other, and CI becomes unreliable. Some teams give up on integration testing entirely, pushing untested transformations straight to production.

4. Compliance and Audit

Regulators and internal auditors ask a simple question: "Show me every data copy operation in the last 90 days." With manual cloning, you cannot answer it. There is no centralized log. There is no record of who ran the clone, when it started, when it finished, which tables succeeded, which failed, or what permissions were applied. The best you can do is search through notebook run histories and hope the clone notebooks have descriptive names. For organizations in regulated industries — finance, healthcare, insurance — this gap is not just inconvenient. It is a compliance violation. Every untracked copy of production data is a potential data governance failure. And the manual response — building a custom audit log on top of an already fragile clone script — adds complexity to a system that was already too brittle to maintain.

What Clone-Xs Does

Clone-Xs is an open-source toolkit that replaces all of the above with a single, governed system. It is not a wrapper around CREATE TABLE ... CLONE. It is a full catalog lifecycle manager that handles cloning, incremental sync, permission copying, tag propagation, validation, rollback, audit logging, and more — across 56 CLI commands, a 31-page web UI, and 61+ REST API endpoints.

Under the hood, Clone-Xs is built as a three-tier application. A React frontend communicates with a FastAPI backend, which delegates to a Python core library of 88 modules. That core library talks to Databricks through the official Databricks SDK and the Statement Execution API, orchestrating every operation against Unity Catalog. There are 12 configuration templates for common scenarios, from nightly staging refresh to PR-specific ephemeral clones.

Key Architectural Insight

All SQL in Clone-Xs flows through a single execute_sql() function with a pluggable executor. When running against a SQL warehouse, it calls the Statement Execution API with automatic retry logic, rate limiting, and polling for long-running queries. In a serverless Databricks notebook, you swap the executor to spark.sql() using set_sql_executor(). One codebase, two compute modes — no code changes required.

The architecture is deliberately modular. The CLI, the web UI, and the notebook integration all share the same Python core. If you fix a bug in the clone logic, it is fixed everywhere. If you add a new feature — say, column-level masking during clone — it surfaces in all three interfaces without duplication.

Feature Deep-Dive: The Clone Wizard

The Clone Wizard is the primary interface for catalog cloning, available both as a 4-step guided flow in the web UI and as a single CLI command. In the UI, the wizard walks you through source and destination selection, schema filtering, clone options, and review. In the CLI, the same operation is a single command with explicit flags.

The 4-Step Flow

Step 1: Source & Destination. Select the source catalog and name the destination. Clone-Xs validates that the source exists and that you have read permissions. If the destination already exists, you are warned and given the option to overwrite, merge, or abort.

Step 2: Schema & Table Filtering. Choose which schemas to include or exclude. Apply regex filters to table names. Tag-based filtering lets you clone only schemas or tables with specific Unity Catalog tags — useful for cloning only "approved-for-dev" data.

Step 3: Clone Options. This is where Clone-Xs differentiates itself. There are 15 configurable options beyond the basic clone type. You control whether to copy permissions, ownership, tags, comments, table properties, CHECK constraints, row-level security, and column masks. You can enable post-clone validation (which compares row counts and schema fingerprints between source and destination), enable rollback (which records every created object so the entire clone can be undone), and enable reporting (which generates a detailed HTML report of every operation).

Step 4: Review & Execute. A summary screen shows the execution plan: how many schemas, tables, views, volumes, and functions will be cloned, estimated time, and estimated cost. You confirm and the clone begins, with a real-time progress bar showing per-schema and per-table status.

CLI Equivalent

The same wizard flow is available as a single CLI command:

clxs clone \
  --source production --dest staging \
  --clone-type DEEP \
  --copy-permissions --copy-ownership --copy-tags \
  --validate --enable-rollback --report

Every flag maps directly to a wizard option. There is no hidden behavior and no magic. The CLI and the UI produce identical results because they both call the same clone_catalog() function in the Python core.

The clone_catalog() function orchestrates the entire flow. It validates inputs, resolves schemas, iterates tables, and delegates each operation to specialized handlers.

def clone_catalog(client, config: dict) -> dict:
    # Step 1: Validate source exists, resolve destination
    validate_catalog_access(client, config["source_catalog"])
    ensure_destination(client, config["destination_catalog"])

    # Step 2: List schemas, apply include/exclude filters
    schemas = list_schemas(client, config["source_catalog"])
    schemas = apply_filters(schemas, config)

    # Step 3: Clone each schema's objects
    for schema in schemas:
        clone_schema(client, schema, config)

    # Step 4: Post-clone validation, permissions, tags
    if config.get("copy_permissions"):
        copy_all_permissions(client, config)
    if config.get("validate"):
        validate_clone(client, config)

Each clone_schema() call handles tables, views, volumes, and functions within that schema, delegating to the pluggable execute_sql() for all DDL operations.

Serverless Mode

For organizations that do not want to provision or pay for a dedicated SQL warehouse for cloning, Clone-Xs supports serverless execution. In this mode, Clone-Xs packages itself as a wheel, uploads it to a Unity Catalog Volume, and submits a Databricks job that runs on serverless compute. Inside the job, it wires spark.sql() as the SQL executor, so every clone operation runs directly on the serverless Spark cluster — no SQL warehouse needed.

clxs clone \
  --source production --dest staging \
  --serverless --volume /Volumes/catalog/schema/vol

The --serverless flag triggers the volume-based deployment. The --volume flag specifies where to upload the wheel and runner script. Clone-Xs handles the rest: building the wheel, uploading artifacts, submitting the job, polling for completion, and streaming logs back to the terminal.

Key Takeaway

The Clone Wizard gives you 15 clone options, post-clone validation, automatic rollback, and HTML reporting — all from a single CLI command or a 4-step UI flow. Serverless mode eliminates the need for a dedicated SQL warehouse entirely.

Feature Deep-Dive: Incremental Sync

Full catalog re-clones are wasteful. If your production catalog has 166 tables across 24 schemas, but only 3 tables changed since the last sync, re-cloning all 166 tables burns compute for no reason. Incremental sync solves this by comparing Delta table versions between the source and destination catalogs.

The mechanism is straightforward. Clone-Xs runs DESCRIBE HISTORY on every table in both the source and destination catalogs, extracts the current Delta version, and compares them. Tables where the source version is ahead of the destination version are flagged for re-sync. Tables that are already current are skipped entirely. The result is a targeted sync that only touches what has changed.

In practice, this reduces clone times dramatically. A full clone of 166 tables might take 45 minutes. An incremental sync of the 3 tables that actually changed takes under 2 minutes. The sync state is persisted to a JSON file between runs, so Clone-Xs remembers which version of each table was last synced and picks up where it left off.

Running Incremental Sync in a Databricks Notebook

For teams that prefer to run sync operations inside Databricks rather than from an external CLI, Clone-Xs provides a clean notebook integration pattern. The key is the set_sql_executor() function, which redirects all SQL execution through spark.sql():

from src.client import set_sql_executor
from src.clone_catalog import clone_catalog
from databricks.sdk import WorkspaceClient

# Wire spark.sql() as the SQL executor
set_sql_executor(lambda sql: [row.asDict() for row in spark.sql(sql).collect()])
client = WorkspaceClient()

result = clone_catalog(client, {
    "source_catalog": "edp_dev",
    "destination_catalog": "edp_dev_00",
    "clone_type": "SHALLOW",
    "load_type": "INCREMENTAL",
    "sql_warehouse_id": "SERVERLESS",
    "copy_permissions": True,
    "copy_tags": True,
    "save_run_logs": True,
})

This pattern is powerful because it means you can schedule incremental syncs as a Databricks Workflow, running on serverless compute, with no external infrastructure. The notebook imports the Clone-Xs library, swaps in the Spark executor, and calls the same clone_catalog() function that the CLI uses. Every feature — permissions, tags, validation, audit logging — works identically.

The incremental sync engine compares Delta versions between source and destination, then re-clones only the tables that have diverged.

def incremental_sync(client, config):
    src_versions = get_table_versions(config["source_catalog"])
    dst_versions = get_table_versions(config["destination_catalog"])

    stale_tables = [
        t for t in src_versions
        if src_versions[t] > dst_versions.get(t, -1)
    ]

    log(f"{len(stale_tables)} tables need re-sync")
    for table in stale_tables:
        execute_sql(f"CREATE OR REPLACE TABLE ... CLONE ...")
    save_sync_state(src_versions)

get_table_versions() runs DESCRIBE HISTORY on each table and extracts the latest Delta version number. The sync state is saved to JSON so subsequent runs can detect new changes.

Key Takeaway

Incremental sync compares Delta table versions and only re-clones what has changed. On a 166-table catalog where 3 tables changed, that turns a 45-minute full clone into a 2-minute targeted sync. The same code runs from the CLI, the API, or a Databricks notebook.

Feature Deep-Dive: Audit Trail to Delta

Every clone operation in Clone-Xs is automatically logged to three Delta tables in a dedicated audit catalog. This is enabled by default — you do not have to configure it, opt in, or remember to turn it on. The moment you run your first clone, Clone-Xs creates the clone_audit catalog, the logs schema, and the three audit tables if they do not already exist.

The Three Audit Tables

Table Purpose Key Columns
run_logs Full execution trace for every job — the complete log output, start/end times, and final status job_id, job_type, status, duration_seconds, started_at, logs_json
clone_operations Operation-level audit trail — who cloned what, when, and the full configuration used operation_id, source_catalog, destination_catalog, user_name, config_json, summary_json
clone_metrics Per-table performance metrics — clone duration, row counts, and byte sizes for capacity planning schema, table, duration_seconds, row_count, size_bytes, success

The clone_operations table has Change Data Feed enabled, which means you can query the history of changes to the audit trail itself. The tags column is a MAP<STRING, STRING> that captures any Unity Catalog tags associated with the operation, enabling tag-based audit queries.

When an auditor asks "show me every data copy operation in the last 90 days," the answer is a single SQL query:

SELECT operation_id, operation_type, source_catalog,
       destination_catalog, status, duration_seconds,
       started_at, user_name
FROM   clone_audit.logs.clone_operations
WHERE  started_at >= DATEADD(DAY, -90, CURRENT_TIMESTAMP())
ORDER BY started_at DESC;

No digging through notebook histories. No scraping Spark UI logs. The data is in a governed Delta table with full lineage, accessible to any BI tool or SQL query.

The audit logger writes structured records to Delta tables after every clone operation completes, regardless of success or failure.

def log_clone_operation(config, result):
    record = {
        "operation_id": str(uuid4()),
        "source_catalog": config["source_catalog"],
        "destination_catalog": config["destination_catalog"],
        "status": result["status"],
        "duration_seconds": result["duration"],
        "user_name": get_current_user(),
        "config_json": json.dumps(config),
    }
    execute_sql(build_insert("clone_audit.logs.clone_operations", record))

The clone_audit catalog and its tables are auto-created on first use. Change Data Feed is enabled on clone_operations so you can track audit trail modifications over time.

Key Takeaway

Clone-Xs automatically persists every operation to three Delta tables: run_logs (execution traces), clone_operations (audit trail), and clone_metrics (performance data). Audit compliance becomes a SQL query, not a forensic investigation.

Feature Deep-Dive: Execution Plan

Sometimes you do not want to clone immediately. You want to see what would happen first. The plan command runs a full dry-run of the clone, captures every SQL statement that would be executed, classifies them by type, and writes them to a file for review.

clxs plan \
  --source production --dest staging \
  --capture-sql plan_statements.sql

The output is a summary of the execution plan, broken down by statement category. For a production catalog with 24 schemas, the output might look like this:

Execution Plan Summary
--------------------------------------
Total statements:  203
CREATE_SCHEMA:      24
CLONE:              166
CREATE_VIEW:        4
CREATE_VOLUME:      9
--------------------------------------
Plan written to: plan_statements.sql

The captured SQL file contains every statement, in execution order, with comments indicating the category and target object. This enables a DBA review workflow that many enterprise teams require: a data engineer generates the plan, a DBA reviews the SQL, approves it (or requests changes), and then the engineer executes the approved plan. This separation of concerns is particularly important for production-to-production clones, where an unreviewed clone could create unexpected objects or grant unintended permissions.

Under the hood, the plan command works by setting a SQL capture hook via set_sql_capture(). The execute_sql() function fires this hook for every statement, even in dry-run mode. The SqlCapture class classifies each statement using pattern matching — a CREATE TABLE ... CLONE is categorized as "CLONE", a CREATE VIEW as "CREATE_VIEW", and so on. The classification logic lives in the classify_sql() function and handles all the standard Unity Catalog DDL patterns.

The SQL capture hook intercepts every statement during a dry run, classifies it, and accumulates a summary report.

class SqlCapture:
    def __init__(self):
        self.statements = []
        self.counts = defaultdict(int)

    def capture(self, sql: str):
        category = classify_sql(sql)
        self.statements.append((category, sql))
        self.counts[category] += 1

# Usage: wire into execute_sql during plan mode
capture = SqlCapture()
set_sql_capture(capture.capture)

classify_sql() uses regex pattern matching to categorize each SQL statement. The captured plan can be written to a file and reviewed before execution.

Key Takeaway

The execution plan command gives DBAs full visibility into every SQL statement before it runs. Generate, review, approve, execute — proper change management for catalog clones, not just "run and hope."

Real-World Scenarios

Nightly Staging Refresh

Your analytics team needs a fresh copy of production in the staging catalog every night. The clone should validate itself after completion, and if validation fails, it should roll back to the previous state automatically. Schedule this as a cron job or a Databricks Workflow:

clxs clone \
  --source production --dest staging \
  --clone-type DEEP \
  --validate --enable-rollback \
  --copy-permissions --copy-tags \
  --report --notify-slack #data-ops

If any table fails validation (row count mismatch, schema difference), the clone is rolled back and Slack is notified. The HTML report is generated regardless, so the morning team can review what happened.

PR-Specific Catalogs for CI/CD

Every pull request should get its own isolated catalog clone for integration testing. The catalog should be created quickly (shallow clone), tagged with the PR number, and automatically cleaned up after a time-to-live expires:

clxs clone \
  --source production --dest ci_pr_1234 \
  --clone-type SHALLOW \
  --ttl 24h

# After tests pass, or after TTL expires:
clxs ttl-cleanup

Shallow clones are nearly instant because they reference the source data files without copying them. The TTL manager runs as a background task and drops expired catalogs automatically. Your CI pipeline creates the catalog, runs tests, and the cleanup happens on its own.

Cross-Workspace Disaster Recovery

Your DR workspace is in a different region. Clone-Xs supports cross-workspace cloning by accepting destination workspace credentials separately from the source:

clxs clone \
  --source production --dest production_dr \
  --dest-host https://dr-workspace.cloud.databricks.com \
  --dest-token $DR_TOKEN \
  --validate --report

The multi-workspace clone module handles authentication to both workspaces, coordinates the clone operation, and validates the result by querying both source and destination. This is the building block for automated DR that actually works.

PII-Safe Development Copies

Before cloning production data to a development catalog, you need to verify that no PII (personally identifiable information) will leak into an environment with relaxed access controls. Clone-Xs includes a PII scanner that checks column names, comments, and sample data for common PII patterns:

# Scan first, then clone only if clean
clxs pii-scan --catalog production

# Or combine scan + clone with auto-masking
clxs clone \
  --source production --dest dev_safe \
  --pii-scan --mask-pii

The PII scanner uses pattern matching on column names (looking for patterns like "ssn", "email", "phone", "date_of_birth") and can optionally sample data to detect PII in values. When combined with the masking module, PII columns are automatically masked in the destination — the original data never leaves the production catalog.

Getting Started

Clone-Xs is designed to be running in under five minutes. Pick the installation method that fits your workflow.

From Source (Recommended for Development)

git clone https://github.com/viral0216/clone-xs.git
cd clone-xs
pip install -e .

This installs Clone-Xs in editable mode, so you can modify the source and see changes immediately. The CLI command clxs is available globally after installation.

Docker (Recommended for Production)

docker compose up -d

This starts both the FastAPI backend and the React UI. The web dashboard is available at http://localhost:3000 and the API at http://localhost:8000. All 31 UI pages and 61+ API endpoints are included.

CLI Only

pip install clone-xs
clxs --help

If you only need the CLI and do not need the web UI or API server, a simple pip install gives you all 56 commands. Configure your Databricks credentials via environment variables (DATABRICKS_HOST, DATABRICKS_TOKEN), a CLI profile (--profile), or interactive browser-based OAuth (clxs login).

Databricks Notebook

For teams that want to run Clone-Xs entirely within Databricks, no external installation is needed beyond uploading the wheel to a Volume. The set_sql_executor() pattern lets you use the full library from any notebook:

# Cell 1: Install the wheel (uploaded to a Volume)
%pip install /Volumes/my_catalog/my_schema/wheels/clone_xs-latest.whl

# Cell 2: Wire spark.sql() and run
from src.client import set_sql_executor
set_sql_executor(lambda sql: [r.asDict() for r in spark.sql(sql).collect()])

from src.clone_catalog import clone_catalog
from databricks.sdk import WorkspaceClient

clone_catalog(WorkspaceClient(), {
    "source_catalog": "production",
    "destination_catalog": "staging",
    "clone_type": "SHALLOW",
    "sql_warehouse_id": "SERVERLESS",
})

This is the same code path that the CLI and API use. No feature is missing, no behavior is different. The notebook is a first-class execution environment, not a second-class wrapper.
